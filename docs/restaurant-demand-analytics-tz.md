# Техническое задание: Аналитическая система прогнозирования спроса и рекомендаций для ресторана

**Проект:** Аналитическая система прогнозирования спроса и рекомендаций для ресторана  
**Версия:** 1.0  
**Дата:** 2026

## 1. Краткое описание проекта

Разработать аналитическую систему, которая на основании исторических данных о продажах за 24 месяца с учетом температуры, календарных событий и внешних трендов формирует:

- прогноз продаж на текущий месяц;
- прогноз продаж на следующий день;
- список необходимых к дозакупке ингредиентов;
- рекомендации по рациону/меню;
- анализ ошибок за предыдущие периоды (план-факт);
- бизнес-рекомендации на основе трендов.

## 2. Цели и задачи

### 2.1 Цель

Создать и внедрить автоматизированную систему анализа и прогнозирования спроса для формирования оптимального производственного плана и подготовки стабильных структурированных данных для загрузки в 1С/IKKO.

### 2.2 Задачи

1. Составить карту данных, необходимых для работы системы.
2. Подобрать инструменты для реализации.
3. Автоматизировать сбор и структуризацию данных.
4. Выполнить анализ продаж прошлых периодов.
5. Провести корреляционный анализ с внешними факторами (температура, праздники, выходные, акции).
6. Анализировать складские остатки для формирования закупочных рекомендаций.
7. Анализировать рыночные тренды.
8. Формировать рекомендации по улучшению меню.
9. Прогнозировать спрос по блюдам на 30 дней и рассчитывать недостающие ингредиенты.
10. Формировать ежедневный отчет с картой заказов и списком недостающих ингредиентов.
11. Выполнять анализ расхождений «план-факт».
12. Генерировать отчеты и рекомендации для менеджмента.
13. Экспортировать стабильные и структурированные данные для последующей загрузки в 1С/IKKO.

## 3. Источники данных

### 3.1 Внутренние данные (обязательные)

#### 3.1.1 Данные продаж

- дата и время продажи;
- название/ID блюда (SKU);
- цена продажи;
- количество;
- данные о температуре в момент продажи (если уже есть в POS, иначе подтягивается из погодного API по timestamp).

#### 3.1.2 Склад и рецептуры

- текущие остатки ингредиентов;
- единицы измерения;
- нормы расхода ингредиентов на блюдо (техкарта/BOM);
- поставщики и минимальные партии закупки (при наличии).

### 3.2 Внешние данные

#### 3.2.1 Погодные данные

Источник: OpenWeatherMap API или Gismeteo API.

- температура;
- осадки;
- архивная погода за прошлые периоды;
- прогноз погоды на 1–30 дней.

#### 3.2.2 Календарно-событийные данные

- государственные праздники;
- локальные городские события;
- собственные маркетинговые акции ресторана;
- признаки дня недели/выходного/предпраздничного дня.

#### 3.2.3 Рыночные тренды

- агрегированные тренды спроса по категориям;
- сезонность;
- внешние индексы интереса (по возможности).

## 4. Функциональные требования

### FR-1. Ежемесячный прогноз продаж

- Система ежедневно пересчитывает прогноз выручки и количества продаж на текущий месяц.
- Выходные метрики: план по дням, категориям и SKU.

### FR-2. Ежедневный прогноз на следующий день

- Формируется прогноз на D+1:
  - общая выручка;
  - прогноз по категориям блюд;
  - прогноз по каждому SKU;
  - часовая нагрузка (карта заказов).

### FR-3. Расчет потребности в ингредиентах и дозакупке

- На основе прогноза по SKU система рассчитывает потребность в ингредиентах через техкарты.
- Формула чистой потребности:

```text
NetPurchase = max((ForecastNeed - CurrentStock) * (1 + BufferPercent), 0)
```

- Буфер по умолчанию: 10% (настраиваемый параметр).
- На выходе: «Заявка на закупку» в формате таблицы (ингредиент, потребность, остаток, к закупке).

### FR-4. Рекомендации по рациону и меню

- Система выявляет:
  - растущие/падающие категории;
  - блюда с высокой маржинальностью и потенциалом роста;
  - блюда-кандидаты для промо, замены или вывода в топ меню.
- Формируются объяснимые рекомендации с причинами (тренд, сезонность, маржа, внешние события).

### FR-5. Анализ «план-факт» и ошибок

- Расчет ошибок по горизонтам D+1 и MTD (month-to-date).
- Метрики:
  - MAPE;
  - WAPE;
  - Bias (систематическое смещение);
  - RMSE (для диагностик).
- Система фиксирует инциденты (значимые отклонения) и предполагаемые причины.

### FR-6. Бизнес-рекомендации

- Генерация коротких управленческих рекомендаций:
  - по закупкам;
  - по операционному планированию кухни;
  - по ассортименту и промо-акциям.

### FR-7. Экспорт в 1С/IKKO

- Ежедневная выгрузка стандартизированного файла с обязательными атрибутами:
  - дата/время;
  - SKU блюда;
  - количество;
  - цена;
  - себестоимость;
  - выручка;
  - признаки погоды/событий.
- Поддерживаемые форматы: CSV и/или JSON (настраивается).

## 5. Нефункциональные требования

- **Достоверность прогноза:** целевой WAPE ≤ 20% на горизонте D+1 через 2 месяца после запуска.
- **Производительность:** обновление ежедневного прогноза и закупочного листа ≤ 10 минут после поступления данных.
- **Доступность:** не менее 99.0% для веб-дашборда в рабочие часы.
- **Безопасность:** ролевая модель доступа (управляющий, шеф-повар, аналитик, администратор).
- **Аудит:** логирование изменений параметров прогноза, буфера и справочников.
- **Наблюдаемость:** мониторинг джобов ETL/ML и контроль качества входных данных.

## 7. Логика аналитики и прогнозирования

### 7.1 Подготовка данных

- Нормализация SKU и ингредиентов.
- Обработка пропусков и выбросов.
- Агрегация до уровней: час, день, категория, SKU.
- Формирование признаков:
  - лаговые признаки (D-1, D-7, D-14, D-28);
  - скользящие средние;
  - день недели, праздник, зарплатный день;
  - погодные признаки (уровень, изменение температуры, осадки);
  - акции и события.

### 7.2 Модели

Рекомендуемый подход:

- Базовая модель: сезонный наивный прогноз (baseline).
- Рабочая модель: градиентный бустинг по табличным данным (например, CatBoost/LightGBM) по SKU и категориям.
- Альтернатива для стабильных рядов: Prophet/ARIMA на уровне категории.
- Ансамблирование при наличии достаточной истории.

### 7.3 Переобучение и контроль качества

- Периодичность переобучения: 1 раз в неделю.
- Пересчет фичей: ежедневно.
- Алертинг при ухудшении качества (рост WAPE > 5 п.п. к медиане последних 4 недель).

## 8. Отчеты и интерфейсы

### 8.1 Основной дашборд

- План/факт за день и месяц.
- Топ отклонений по категориям и SKU.
- Влияющие факторы (погода/события/акции).

### 8.2 Ежедневный прогноз (D+1)

- Прогноз выручки.
- Прогноз по категориям и SKU.
- Почасовая карта заказов.
- Кнопка «Сформировать заявку на закупку».

### 8.3 Закупочный лист

- Список ингредиентов с расчетом:
  - прогнозная потребность;
  - текущий остаток;
  - буфер;
  - итог к закупке.

### 8.4 План-факт анализ

- График ошибок и инцидентов.
- Причинно-следственные комментарии системы.
- Рекомендации по улучшению точности.

## 9. Интеграции

### 9.1 Интеграция с погодными API

- OpenWeatherMap или Gismeteo.
- Механизм fallback при недоступности источника.
- Кэширование ответов для снижения стоимости запросов.

### 9.2 Интеграция с 1С/IKKO

- Выгрузка ежедневного файла в согласованный каталог/endpoint.
- Валидация схемы файла перед отправкой.
- Повторная отправка при ошибке доставки.

## 10. Ключевые сценарии использования

Сценарий пользователя (управляющая Анна):

1. Открывает дашборд и видит уведомление о готовом прогнозе.
2. На вкладке «Ежедневный прогноз» анализирует ожидаемую выручку и факторные влияния (погода/события).
3. Запускает формирование закупочного листа.
4. Отправляет поставщику список к дозакупке.
5. Использует почасовую карту заказов для планирования смены кухни.
6. Проверяет раздел «План-факт» за прошлую неделю и принимает управленческие решения.
7. В конце дня система автоматически формирует файл для 1С/IKKO.

## 11. KPI и критерии приемки

### 11.1 KPI

- Точность D+1: WAPE ≤ 20%.
- Доля дней с критическим stockout по топ-20 ингредиентам: ≤ 5%.
- Сокращение списаний: минимум на 10% через 3 месяца.
- Сокращение ручного времени на планирование закупок: минимум на 50%.

### 11.2 Критерии приемки

- Полный ETL-процесс от загрузки данных до отчета выполняется автоматически.
- Ежедневный прогноз и закупочный лист формируются без ручного вмешательства.
- Сформированные данные принимаются системой 1С/IKKO без ошибок схемы.
- В интерфейсе доступны объяснения ключевых факторов прогноза.

## 12. Этапы реализации

1. **Discovery и карта данных (2–3 недели):** аудит источников, согласование справочников.
2. **ETL и витрины (3–4 недели):** сбор, очистка, унификация данных.
3. **MVP-прогноз и закупки (4–6 недель):** baseline + рабочая модель, ежедневные отчеты.
4. **План-факт и рекомендации (2–3 недели):** аналитика отклонений и бизнес-советы.
5. **Интеграция с 1С/IKKO (2 недели):** формат обмена, тестовые загрузки.
6. **Пилот и донастройка (4 недели):** A/B-сравнение, оптимизация модели.

## 13. Риски и меры

- **Низкое качество исходных данных:** внедрить контроль качества, валидации и справочники.
- **Неполная событийная календаризация:** вести локальный календарь событий и обратную связь от управляющего.
- **Недоступность внешних API:** fallback-источники и кэш.
- **Дрейф спроса:** регулярное переобучение и мониторинг ошибок.

## 14. Рекомендуемый состав выходных артефактов

- Веб-дашборд (роль управляющего/шефа/аналитика).
- Ежедневный отчет D+1 (PDF/HTML + таблицы CSV).
- Файл обмена для 1С/IKKO.
- Лог качества моделей и отчет plan-fact.
- Реестр рекомендаций и принятых управленческих решений.
